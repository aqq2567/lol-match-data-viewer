## 1. 产品概述

### 1.1 产品定位

**LOL Match Data Viewer** 是一款基于 Electron 的《英雄联盟》对局数据分析桌面应用。通过连接 LCU（League Client Update）本地 API，实时获取玩家对局数据，提供深度统计分析、多维度排名和 AI 辅助洞察。

不同于网页端战绩查询工具（如 OP.GG），本应用：
- 运行在本地，无需网络请求第三方服务
- 数据源为 LCU 本地 API，可获取完整对局详情（含装备时间线、符文、海克斯增幅等 170+ 维度）
- 专为硬核玩家设计，提供跨多场的聚合分析而非单场摘要

### 1.2 核心价值主张

| 痛点 | 解决方案 |
|------|---------|
| 客户端只显示最近 20 场，无聚合视图 | 拉取全部 ~200 场，按模式筛选 + 聚合分析 |
| 想对比不同队友的数据表现 | 按玩家维度聚合：KDA、经济、输出、承伤等 10+ 指标排名 |
| 不知道自己的英雄池/出装模式 | 英雄频率、装备频率、海克斯频率 自动统计 |
| 想量化"谁是车队大腿" | 多场聚合后按指标排序，可视化领奖台 TOP3 |
| 看不懂原始数据 | AI 助手注入全部对局数据，自然语言问答 |

## 2. 目标用户画像

| 用户类型 | 特征 | 核心诉求 |
|----------|------|---------|
| **竞技玩家** | 单排/组排为主，关注胜率和 KDA | 分析自己的强弱项指标，找提升方向 |
| **车队队长** | 固定 5 人车队，需要评估队员表现 | 对比队友数据，量化贡献度 |
| **英雄专精玩家** | 只玩 1-3 个英雄 | 查看英雄在不同对局中的表现模式 |
| **数据爱好者** | 喜欢看数据、做分析 | 多维指标探索，AI 深度问答 |

## 4. 非功能性需求

### 4.1 性能

| 指标 | 目标 | 当前状态 |
|------|------|---------|
| 对局列表加载（200场摘要） | < 5s | ✅ 6 次并行分页 |
| 详情批量加载（200场详情） | < 15s | ✅ 并发 30，分批 |
| 分析计算（200场聚合） | < 1s | ✅ 纯函数，同步计算 |
| 图片加载 | 不阻塞 UI | ✅ 异步 + 并发限制 8 |
| 应用启动（冷启动） | < 3s | ✅ |
| 内存占用（200场缓存） | < 500MB | 待验证 |

### 4.2 可靠性

| 场景 | 处理策略 | 状态 |
|------|---------|------|
| LCU 未连接 | 显示"等待客户端连接"空状态 | ✅ |
| LCU 中途断开 | 下次操作时检测并提示 | ✅ |
| API 请求超时 | 120s 超时 + 错误提示 | ✅ |
| 自动更新 GitHub 不可达 | 镜像回退（ghproxy） | ✅ |
| AI API Key 未配置 | 使用内置默认 Key | ✅ |
| 不同模式对局混合选择 | 弹窗警告阻止分析 | ✅ |

### 4.3 安全

| 要求 | 实现 |
|------|------|
| API Key 不入代码 | `.env` + `.gitignore` |
| 日志不含敏感信息 | 手动审查 logger 过滤 |
| 渲染进程不直接访问文件系统 | Electron `contextIsolation: true` |
| 主进程-渲染进程通信走 IPC | `contextBridge` + `ipcRenderer.invoke` |

### 4.4 兼容性

| 平台 | 支持状态 |
|------|---------|
| Windows 10/11 | ✅ 主要目标 |
| Windows 7/8 | 理论上支持，未测试 |
| macOS | ❌ 未适配 |
| 国服 | ✅ 适配 `begIndex`/`beginIndex` 双拼写 |
| 外服 | 理论上支持，未测试 |

## 5. 技术架构约束

### 5.1 层级依赖规则

```
表示层 (Vue SFC) → 应用层 (services/ports) → 领域层 (纯函数)
                       ↓
                  基础设施层 (Electron main + preload)
```

- 领域层：零外部依赖，纯 TypeScript 标准库
- 应用层：通过 Port 接口访问基础设施，不 import 具体实现
- 表示层：通过适配器工厂（`createXxxRepository(window.lcuApi)`）获取 port 实例
- 基础设施层：实现 IPC handler，暴露 `window.lcuApi`

### 5.2 数据源上限

| 数据源 | 上限 | 说明 |
|--------|------|------|
| LCU match-history API | ~200 场 | Riot 服务端硬限制 |
| SGP API（计划中） | 无限制 | 需要 Bearer JWT 认证 |

### 5.3 技术债务（来自架构扫描）

| # | 问题 | 影响 | 优先级 |
|---|------|------|--------|
| 1 | `extractor.ts` 925 行上帝模块 | 新字段添加困难，难以测试 | P1 |
| 2 | `FriendAnalysis.vue` 819 行上帝组件 | 与 AnalysisView 共享 90% 逻辑但未复用 | P1 |
| 3 | 测试覆盖率极低（仅 1 个测试文件） | 回归风险高 | P1 |
| 4 | Python 工具复制了 TS 提取逻辑 | 字段变更需双端同步 | P2 |
| 5 | 领域层直接依赖 `@shared/utils/mappings` | 层级泄漏（应走应用层） | P2 |
| 6 | Port 适配器每次临时创建，非 DI 注入 | 不利于单元测试 mock | P2 |
| 7 | `tableMaxHeight` 公式 2 处重复 | 维护不一致风险 | P3 |
| 8 | `shortName()` 3 处重复 | 轻微代码异味 | P3 |

## 6. 功能详情

### 6.1 多场分析模式

用户从对局列表选中 N 场同模式对局 → 分析页展示：

```
┌─────────────────────────────────────────────────┐
│ AnalysisHeader: 模式名 | 队友开关 | W/L/胜率卡片  │
├──────────┬──────────────────────────────────────┤
│ Metric   │ Podium TOP3                          │
│ Sidebar  │ RankingTable（玩家排名）               │
│          │                                      │
│ 基础指标  │ Detail（英雄池/装备/增幅/高阶指标）      │
│ ├KDA     │                                      │
│ ├击杀    │ ChatPanel（AI 助手）                   │
│ ├死亡    │                                      │
│ └助攻    │                                      │
│ 高阶指标  │                                      │
│ ├MVP分   │                                      │
│ └...     │                                      │
└──────────┴──────────────────────────────────────┘
```

**模式自适应**：不同模式显示不同指标集（`mode-analysis-config.ts` 定义）。

### 6.2 好友分析

选中某个好友 → 显示：
- 共玩场次、胜率、被好友邀请频率
- 与好友同队时的数据 vs 全局数据对比
- 按好友拆分的详细指标

### 6.3 AI 对话

- 对局数据格式化为结构化文本注入 system prompt
- 模块级 `ref` 保证切换指标时消息不丢
- game_id 指纹检测切换对局时清空对话
- Chat completions 模式（非流式），120s 超时

## 7. 术语表

| 术语 | 说明 |
|------|------|
| LCU | League Client Update — LOL 客户端内置的本地 HTTP API |
| SGP | Service Gateway Proxy — Riot 后端直连 API |
| Port | 应用层定义的抽象接口，解耦业务逻辑与基础设施 |
| Adapter | Port 的具体实现，将 `window.lcuApi` 方法名映射为业务语义 |
| 上帝模块 | 单文件承载过多职责，违反单一职责原则 |
| KeepAlive | Vue Router 缓存机制，切换页面时保留组件状态 |
